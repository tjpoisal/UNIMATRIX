# Unimatrix — MVP Scope

**Last updated:** 2026-05-11 (v2 — revised after Gemini technical review)

---

## The one thing the MVP must prove

> When you open any AI client on any device, it instantly knows where you left off — zero re-establishment friction.

Everything that doesn't serve that proof is Phase 2 or later.

---

## What we are building

A cloud-hosted MCP server backed by a two-layer memory engine and an async Librarian agent. No mobile app. No full web portal. One integration target: Claude Desktop — which, because MCP configurations are now account-linked, automatically unlocks Claude iOS and Android as well.

---

## Memory architecture — two layers

Confirmed correct by Gemini review. Both layers are required from day one.

### Layer 1 — Episodic memory (verbatim)
- Append-only, chronological log of raw inputs exactly as received
- Never modified after write; only ever tagged with `superseded_at` timestamps
- Queried only for: audit trails, source attribution, and Temporal Replay ("show me what I knew on date X")
- Lives in the `memories` table, `content` column

### Layer 2 — Semantic memory (living synthesis)
- Continuously updated compressed abstractions — what is currently true
- What `recall_context` queries by default
- Generated by the Librarian agent each time related memories are added or superseded
- Lives in the `memories` table, `summary` column + `embedding` vector
- The LLM never sees the raw episodic log unless explicitly asked

**Rule:** `recall_context` hits Layer 2 (semantic). `get_timeline` hits Layer 1 (episodic). Never mix them in the same payload.

## Build phases (10 weeks)

### Phase 0 — Foundation (Weeks 1–2)
- Supabase project setup with pgvector extension
- Schema migrations (see schema section below)
- Clerk auth integration + RLS policies (users access only their own data)
- Fastify API scaffold with Zod validation
- Railway deploy pipeline + CI
- Encryption baseline: application-layer encrypt `content` column before write (see encryption section)

### Phase 1 — Memory engine (Weeks 3–4)
- Memory CRUD API endpoints
- Vector embeddings generated on write — **Voyage AI (Voyage Multimodal 3.5)** as primary; evaluate vs. text-embedding-3-small on recall accuracy before committing
- Hybrid search on `recall_context`: cosine similarity + exponential recency decay (current facts weighted heavily over superseded ones)
- Hard token budget on `recall_context` payload: 4,000 tokens desktop, 800 tokens mobile/cellular
- Space auto-creation logic
- Time-aware status flags: `active`, `superseded`, `archived`

### Phase 2 — Librarian agent (Weeks 5–6)
- LLM classification pipeline using Claude Haiku (~$0.0005 per classification)
- Auto-tag and route memories into Spaces
- **Async jobs via Inngest or Trigger.dev** (replaces BullMQ + Redis — serverless, built-in retry, zero Redis infra to manage)
- **Poly-tagging for ambiguous routing:** if confidence < 85%, route memory to all candidate Spaces. Pointer-level duplication is cheap; a review queue is a UX graveyard.
- Non-blocking: `store_memory` always returns immediately

### Phase 3 — MCP server (Weeks 7–8)
- 4 core MCP tools (see below)
- Claude Desktop `claude_desktop_config.json` integration
- Account-linked MCP config automatically available on Claude iOS + Android — no separate mobile bridge needed
- Desktop wake-up profile: 4,000-token rich context payload
- Mobile/cellular profile: 800-token concise payload (detected by client hint or user agent)
- End-to-end Return to Context (RTX) test and benchmark
- Rate limiting + per-user auth on MCP endpoints

### Phase 4 — Ship and learn (Weeks 9–10)
- Minimal management UI: view, search, edit, delete memories
- Onboarding flow: connect MCP → store first memory → experience RTX
- Waitlist / beta signup page
- Usage analytics (Mixpanel or Supabase logging)
- Goal: 5 beta users active daily for 2 consecutive weeks

## The 4 MCP tools — nothing more for MVP

| Tool | Signature | Layer queried | Purpose |
|------|-----------|---------------|---------|
| `store_memory` | `(content: string, hint?: string)` | Write | Drops raw content into Librarian queue. Non-blocking. Always returns immediately. |
| `recall_context` | `(topic?: string, limit?: number)` | Semantic (L2) | Hybrid search: semantic similarity + recency decay. Token-budgeted payload. The RTX moment. |
| `search_memories` | `(query: string)` | Semantic (L2) | Explicit semantic search when recall_context isn't targeted enough. |
| `get_timeline` | `(topic: string)` | Episodic (L1) | Chronological verbatim view with invalidation markers. Temporal Replay preview — premium bait. |

---

## `recall_context` — hybrid search spec

Pure cosine similarity is a trap: it surfaces semantically similar but stale or superseded facts equally with current ones. The retrieval function must be:

```
score = cosine_similarity(query_embedding, memory_embedding)
      × exp(-λ × days_since_created)
      × (status == 'active' ? 1.0 : 0.1)
```

- `λ` (decay rate): start at 0.01 (half-life ~70 days), tune against beta user feedback
- Superseded memories get a 10× penalty — they stay retrievable for Temporal Replay but don't pollute `recall_context`
- Hard cap: trim to token budget before returning (most relevant first)

## Librarian — poly-tagging spec

When the Librarian Haiku model's confidence is split across multiple Spaces:

| Confidence | Action |
|------------|--------|
| ≥ 85% for one Space | Route to that Space only |
| < 85%, two candidates | Route to both Spaces (soft link — pointer duplication) |
| < 85%, 3+ candidates | Route to top 2 by confidence score |
| No confident match | Create a new Space named after the dominant topic |

**Never quarantine.** A memory stuck in a review queue is a memory that never gets recalled. Zero-friction filing is a core product promise.

## Hierarchy — flatten it for MVP

**Spec design (too deep for now):**
```
Wing → Hall → Room → Drawer
```

**MVP design:**
```
Space (Librarian assigns)
  └── Memory + tags (Librarian assigns)
```

Two levels. Users never navigate a hierarchy to drop a note. The full 4-level design becomes a schema migration in Phase 2 once we understand how users actually organize their palace.

Design the `spaces` table with a `parent_id` nullable column from day one — the Phase 2 migration is then just flipping a nullable column, not a schema rewrite.

## Database schema (MVP)

```sql
-- Users (managed by Clerk, mirrored here for FK)
CREATE TABLE users (
  id UUID PRIMARY KEY,
  clerk_id TEXT UNIQUE NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Spaces (2-level hierarchy for MVP; parent_id ready for Phase 2 depth)
CREATE TABLE spaces (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  parent_id UUID REFERENCES spaces(id),  -- nullable; ready for deeper hierarchy
  org_id UUID,                            -- nullable; wired up in Phase 3 for team palaces
  name TEXT NOT NULL,
  description TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Memories
CREATE TABLE memories (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  space_id UUID REFERENCES spaces(id),
  org_id UUID,                              -- nullable; Phase 3 team palaces
  content BYTEA NOT NULL,                   -- application-layer encrypted ciphertext (AES-256-GCM)
  content_iv BYTEA NOT NULL,                -- initialization vector for decryption
  summary TEXT,                             -- Librarian-generated synthesis (plaintext OK — less sensitive)
  status TEXT DEFAULT 'active',             -- active | superseded | archived
  superseded_by UUID REFERENCES memories(id),
  superseded_at TIMESTAMPTZ,
  confidence FLOAT,                         -- Librarian routing confidence
  embedding VECTOR(512),                    -- MRL-compressed Voyage AI embedding (512d saves ~66% storage vs 1536d)
  tags TEXT[],
  source TEXT,                              -- 'mcp' | 'api' | 'import'
  created_at TIMESTAMPTZ DEFAULT NOW(),
  indexed_at TIMESTAMPTZ                    -- when Librarian finished processing
);

-- Row Level Security (enforced at DB level — users access only their own data)
ALTER TABLE memories ENABLE ROW LEVEL SECURITY;
CREATE POLICY memories_isolation ON memories
  USING (user_id = auth.uid());

ALTER TABLE spaces ENABLE ROW LEVEL SECURITY;
CREATE POLICY spaces_isolation ON spaces
  USING (user_id = auth.uid());

-- Indexes
CREATE INDEX ON memories USING ivfflat (embedding vector_cosine_ops);
CREATE INDEX ON memories (user_id, status);
CREATE INDEX ON memories (space_id, created_at DESC);
CREATE INDEX ON memories (user_id, created_at DESC);  -- recency decay query
```

**Schema notes:**
- `content` is `BYTEA` not `TEXT` — designed for ciphertext from day one. Retrofitting encryption onto a live `TEXT` column is a painful migration.
- `embedding` is `VECTOR(512)` not `VECTOR(1536)` — Voyage AI with MRL support lets us compress to 512 dimensions with minimal accuracy loss, saving ~66% pgvector storage.
- `org_id` is nullable on both tables now — Phase 3 just populates it and adds the FK constraint.

## Tech stack

### Core
- **Runtime:** Node.js 20+ with TypeScript (strict mode)
- **API framework:** Fastify + Zod
- **MCP server:** `@modelcontextprotocol/sdk`
- **Librarian LLM:** Claude Haiku (~$0.0005 per classification)

### Data
- **Database:** Supabase (managed PostgreSQL + pgvector + RLS)
- **Async jobs:** Inngest or Trigger.dev (replaces BullMQ + Redis)
- **Embeddings:** Voyage AI Voyage Multimodal 3.5 with MRL (evaluate vs. text-embedding-3-small before committing — run both on a 200-query benchmark)

### Infra and auth
- **Hosting:** Railway
- **Auth:** Clerk
- **Secrets:** Doppler or Railway env vars
- **Encryption keys:** Per-user AES-256-GCM keys, stored in Railway secrets or AWS Secrets Manager (not in the DB)

## Encryption at rest — tiered strategy

| Tier | Approach | What it protects against |
|------|----------|--------------------------|
| MVP / Beta | Supabase default AES-256 + application-layer encrypt `content` column | Supabase infrastructure breach; DB dumps |
| Pro | Per-user encryption keys in Secrets Manager | Compromised DB + app server; Supabase team access |
| Enterprise | BYOK (Bring Your Own Key) via AWS KMS or HashiCorp Vault | All of the above + you never hold customer keys |
| Self-hosted | User runs their own Supabase instance | Full sovereignty; zero trust required |

**Implementation for MVP:**
1. On `store_memory`: generate `iv` → encrypt `content` with user's key → store `BYTEA` ciphertext + `iv`
2. On retrieval: fetch ciphertext + iv → decrypt in application layer → return plaintext to MCP caller
3. Embeddings are generated from plaintext *before* encryption — the vector index never needs to touch ciphertext
4. `summary` (Librarian synthesis) is stored as plaintext — it's an abstraction, not raw sensitive content. Encrypt in Pro tier if needed.

**Do not skip RLS.** Even with application-layer encryption, row-level security is the defense-in-depth layer that prevents a bug in the API from leaking another user's encrypted blobs.

## MVP success criteria

The MVP is done when all four are true:

1. `recall_context` returns relevant memories in **< 3 seconds** from a cold Claude Desktop session
2. Librarian correctly routes memories to the right Space with **> 80% accuracy** on a test set of 50 varied inputs
3. All **4 MCP tools** work end-to-end in Claude Desktop (and Claude iOS via account-linked MCP) without errors
4. **5 beta users** actively using it daily for 2+ consecutive weeks without being prompted
